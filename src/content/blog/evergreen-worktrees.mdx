# Evergreen Worktrees for Coding Agents

Claude Code has good support for git worktrees. You can start a session with
`claude --worktree feature-auth`, and it will create an isolated checkout under
`.claude/worktrees/<name>` on its own branch. The docs also cover copying
gitignored files with `.worktreeinclude`, isolating subagents in temporary
worktrees, and cleaning them up when they are no longer needed.

That solves the file-collision problem. It does not solve the full-stack
environment problem.

My app is a TypeScript React Router app with an Express server, Prisma,
Postgres, Docker Compose, Mailpit for local email, and MinIO for S3-compatible
storage. A fresh checkout is not immediately useful. It needs dependencies,
generated Prisma clients, migrations, seed data, service containers, local
ports, and usually a browser session pointed at the right server.

When an AI coding agent creates a new worktree from scratch, it can spend a
surprising amount of the session rediscovering that environment. Worse, two
agents that both try to run "the app" will usually fight over `localhost:3000`,
Postgres, Mailpit, or whatever the project assumes is globally unique.

So I stopped treating worktrees as disposable checkouts and started treating
them as a small pool of long-lived dev environments.

I call them evergreen worktrees.

## What I mean by evergreen

My checkout looks roughly like this:

```text
prepa-nav-web/
prepa-nav-web-wt1/
prepa-nav-web-wt2/
prepa-nav-web-wt3/
prepa-nav-web-wt4/
prepa-nav-web-wt5/
```

These are ordinary git worktrees. There is no magic there. The difference is
that each one is expected to stick around across sessions. It has its own
branch, its own local `.env.local`, its own Docker Compose project state, its
own Postgres volume, its own MinIO volume, and its own assigned ports.

The port assignment is deterministic. The main checkout is index `0`; sibling
worktrees are named `prepa-nav-web-wtN`.

```text
PORT               = 3000 + N
DB_PORT            = 5434 + N
MAILPIT_SMTP_PORT  = 1025 + N
MAILPIT_UI_PORT    = 8025 + N
MINIO_PORT         = 9000 + N
MINIO_CONSOLE_PORT = 9100 + N
```

So `wt2` gets the app on `3002`, Postgres on `5436`, Mailpit UI on `8027`, and
MinIO on `9002`. The exact numbers do not matter. What matters is that no agent
has to invent them, and no two worktrees claim the same resources.

Each worktree has a local file like this:

```dotenv
PORT=3002
DB_PORT=5436
MAILPIT_SMTP_PORT=1027
MAILPIT_UI_PORT=8027
MINIO_PORT=9002
MINIO_CONSOLE_PORT=9102
DATABASE_URL="postgresql://pa_apply:pa_apply@localhost:5436/pa_apply?schema=public"
TEST_DATABASE_URL="postgresql://pa_apply:pa_apply@localhost:5436/pa_apply_test?schema=public"
AWS_ENDPOINT_URL_S3="http://localhost:9002"
BUCKET_NAME=pa-apply-local
AWS_ACCESS_KEY_ID=localdev
AWS_SECRET_ACCESS_KEY=localdev
```

This file is not a secrets file. It is a collision-avoidance file. It contains
local ports, local database URLs, and fake local object-storage credentials.

The app's dev command then brings up services for that worktree and starts the
server:

```sh
doppler run -- docker compose --env-file .env.local up -d --wait
doppler run -- cross-env NODE_ENV=development MOCKS=true node ./server/dev-server.js
```

In practice that is wrapped as `npm run dev`, but the important detail is that
Docker Compose reads the per-worktree `.env.local`, while shared application
configuration is injected separately through an allowlisted project command.

## Why this helps agents

The most obvious benefit is parallelism. I can have one agent working on a
program details page in `wt4`, another debugging an extraction script in `wt2`,
and another running browser checks against `wt1`. They can each start the app,
send OTP emails to their own Mailpit instance, upload files to their own MinIO
bucket, and run database-backed tests without trampling each other.

The less obvious benefit is that the environment becomes boring.

I do not want the agent spending attention on questions like:

- Which port should I use?
- Is Postgres already running?
- Did this worktree get seeded?
- Where is Mailpit for this checkout?
- Is this `.env` file real or stale?
- Did another agent just kill my dev server?

Those are not interesting software engineering questions. They are local
plumbing. If the agent has to solve them repeatedly, I have made the project
harder for the agent than it needs to be.

Evergreen worktrees turn that plumbing into a small convention:

- `wtN` implies a fixed set of ports.
- `.env.local` contains only values that must differ per worktree.
- Docker services are local to the worktree.
- The database can be migrated and seeded once, then reused.
- Browser automation gets a session name derived from the worktree.

This also makes handoff easier. If I say "use `wt3` for this", that means a
branch, a server port, a database, an email UI, an object store, and a browser
session namespace. It is more like handing someone a small VM than handing them
a copy of the source tree.

## The boring part is the product

The useful piece is not just "use git worktrees." It is the runbook around them.

I have agent-readable skills for three operations:

- `create-worktree`: create `../prepa-nav-web-wtN` from `dev`, bind it to the
  agent-safe dev secret config, write `.env.local`, install dependencies,
  generate Prisma, start Docker, run migrations, and seed data.
- `validate-worktrees`: read-only health check across all worktrees: branches,
  Doppler binding, `.env.local`, port uniqueness, symlinks, and Docker status.
- `refresh-worktrees`: repair local config drift, backfill missing local keys,
  re-run dependency install, and avoid destructive branch or database resets
  unless explicitly requested.

This is the part I would copy into another serious project. The scripts are not
complicated, but the policy matters:

- Worktree creation is sequential, because port assignment has a race.
- Branches start from `dev`, not whatever branch the main checkout happens to be
  on.
- `.env.local` is never synced wholesale across worktrees.
- Existing local overrides are reported instead of overwritten.
- Destructive branch resets and database reseeds require explicit human intent.

Agents are pretty good at following local conventions when those conventions are
written down in the repo. They are much worse when the convention only exists in
my head.

## The security trap

The easy version of this setup is dangerous.

Claude Code's worktree docs mention `.worktreeinclude` for copying gitignored
files such as `.env` into fresh worktrees. That is convenient, but it is the
wrong shape for my threat model. I do not want secret-bearing files copied into
agent worktrees at all.

I use Doppler for shared configuration. That is better than checking in `.env`,
but it does not automatically make agent execution safe. If an agent can run
arbitrary Doppler CLI commands with my personal auth, it can ask Doppler to
print secrets. If an agent can run `doppler run -- env`, it can dump everything
the app would receive. A hook that says "Doppler commands are approved" is not a
security boundary.

The setup I use for agent-accessible worktrees is stricter:

- Human shells use my normal Doppler login. Agent shells do not.
- Agents only get access to a separate `agent-dev` config with fake or
  low-blast-radius values.
- That config is exposed to the command runner through a read-only Doppler
  service token scoped to a single project/config, not through my personal CLI
  token.
- The token is not enough by itself. The command surface is also restricted:
  agents can run allowlisted project commands like `npm run dev`, `npm test`,
  and `npm run seed`, but not `doppler secrets get`, `doppler secrets download`,
  token management commands, or arbitrary shells under `doppler run`.
- No staging or production secrets are available from agent worktrees.
- `.env.local` remains local plumbing only: ports, local DB URLs, Mailpit, and
  fake MinIO credentials.

The key distinction is that "the app can receive secrets at runtime" and "the
agent can inspect secrets" are different capabilities. A read-only service token
narrows the blast radius, but it does not prevent exfiltration if the agent can
freely choose the command being wrapped. You need both least-privilege values
and a narrow execution interface.

I would expect this paragraph to be the first thing people poke at, and they
should. If your agent can run arbitrary shell commands with your normal cloud
credentials, worktree isolation is not doing much for you.

## What gets worse

Evergreen worktrees are not free.

They use more disk. They use more Docker resources. They can accumulate stale
branches. Their databases can drift from seed state. A bug can disappear because
`wt2` happens to have old rows that `wt4` does not. Node dependencies can fall
behind after a package-lock change. If you are not careful, the whole thing
turns into local snowflake infrastructure.

The mitigation is to make drift visible.

That is why I have a read-only validation command. It checks whether each
worktree has the expected Doppler binding, expected ports, no stray `.env`,
valid symlinks for local agent config, and Docker services in the expected
state. The refresh command repairs additive drift, but it does not wipe local
branches or databases unless I explicitly ask it to.

I still rely on CI. I still expect final validation on a clean-ish branch. The
evergreen worktree is not an authority. It is a warm place for exploratory and
implementation work.

## When I would not do this

I would not bother with this for a library, a static site, or a small app where
`npm install && npm test` is the whole environment.

I also would not use it if the team already has cheap disposable cloud dev
environments. A per-PR environment with real isolation, automatic teardown, and
safe secret injection is better than a pile of local folders if the cost and
latency are acceptable.

Evergreen worktrees make sense when:

- the app is stateful enough that setup time matters;
- multiple agent sessions often run at once;
- local Docker services are part of normal development;
- browser checks need a stable URL;
- the project has enough conventions that agents benefit from a prepared
  workspace.

The pattern is basically "pet dev environments for disposable branches." That
sounds backwards, but it matches how I actually use coding agents. The task is
temporary. The environment does not have to be.

## The punchline

The thing that made AI coding agents more useful for me was not a clever prompt.
It was reducing the amount of local environment discovery they had to do.

Git worktrees gave me file isolation. Evergreen worktrees gave me environment
isolation with names I can remember, ports that do not collide, seeded databases
that are already warm, and runbooks the agents can read.

This is probably overkill for many projects. It is also not a substitute for
real sandboxing or CI. But for a stateful full-stack app, it has made agents
less expensive to supervise.

That is a good enough reason for me to keep the worktrees alive.
