# CLAUDE.md

Guidance for Claude Code when working in this repository.

## Stack

- Astro 5 (static output, no SSR)
- Tailwind CSS v4 via `@tailwindcss/vite`
- MDX for blog posts (`@astrojs/mdx`)
- TypeScript
- Vanilla `<script>` blocks for interactivity (no React)
- Deployed on Cloudflare Pages (auto-deploy from `main`)

## Commands

```bash
npm install           # install deps
npm run dev           # dev server on :4321 (regenerates sprite first)
npm run build         # static output in dist/
npm run preview       # serve dist/ locally
npm run build:icons   # rebuild public/sprite.svg from icons/
npm run sync-books    # populate src/data/books.json, mirror covers to R2
npm run sync-photos   # populate src/data/photos.json, mirror photos to R2
npm run sync-homelab  # populate src/data/homelab.json from Homepage services.yaml
```

## Bookshelf data flow

The `/bookshelf` page reads `src/data/books.json`. Cover images live in the R2
bucket `personal-site-photos` served at `https://media.about-clay.com/covers/{id}.jpg`
вЂ” not committed to the repo. `scripts/sync-books.mjs` pulls a curated Hardcover
list (by slug), uploads any new covers to R2 via `rclone rcat`, orphan-cleans
covers whose IDs have left the list, and writes a `coverUrl` per book into the
metadata JSON.

Flags:
- `--dry-run`  write books.json locally; skip R2 uploads and git
- `--no-push`  commit but do not push

Env (all required):
- `HARDCOVER_API_KEY`, `HARDCOVER_LIST` (see `scripts/lib/hardcover.mjs`)
- `R2_PHOTOS_REMOTE`, `R2_PHOTOS_PUBLIC_URL`

The script compares new JSON against the committed copy before committing, and
pushes with a bot identity.

On the server (`cloud-hil-1`), the script runs daily via
`personal-site-sync-books.timer`. The repo is cloned at `/opt/personal-site`;
push auth is a per-repo SSH deploy key at `/root/.ssh/personal_site_deploy`
(write access) used via the `github-personal-site` Host alias in
`/root/.ssh/config`.

## Photos data flow

The `/photos` page reads `src/data/photos.json`. Asset bytes are *not* committed вЂ”
they live in R2 bucket `personal-site-photos`, served at `https://media.about-clay.com`.
`scripts/sync-photos.mjs` fetches a curated Immich album (by UUID), mirrors new
web-sized JPEGs to R2 via `rclone rcat`, orphan-cleans R2 objects no longer in the
album, and writes the metadata JSON.

Env vars on cloud-hil-1: `IMMICH_API_KEY`, `IMMICH_API_URL`, `IMMICH_ALBUM_ID`,
`R2_PHOTOS_REMOTE`, `R2_PHOTOS_PUBLIC_URL`. Runs daily via
`personal-site-sync-photos.timer`.

## Homelab data flow

The `/homelab` page reads `src/data/homelab.json` (services list) plus static
meta (`host`, `blurb`) hard-coded in `src/data/homelab.ts`.
`scripts/sync-homelab.mjs` parses the Homepage dashboard's `services.yaml`
(at `$HOMEPAGE_CONFIG_DIR/services.yaml` on cloud-hil-1) and writes the JSON.

Only services explicitly flagged `public: true` in the YAML are exported to
the site вЂ” default is to stay private. The sync also honors two custom keys
Homepage itself ignores: `upstream: <url>` (shown as a secondary "upstream в†’"
link) and `tags: [..]` (rendered as a tech-stack row). Sections become
`category`.

Env on cloud-hil-1: `HOMEPAGE_CONFIG_DIR` (required). Expected to run daily
via `personal-site-sync-homelab.timer`.

### Adding a new dynamic content source

To add another external content source (e.g., photos from Immich, podcasts):

1. Add `scripts/sync-X.mjs` modeled on `sync-books.mjs`: fetch from source в†’
   write `src/data/X.json` (+ assets in `public/X/` if applicable).
2. Diff new JSON against the committed copy before writing; orphan-clean
   assets whose IDs are no longer in the new set.
3. Commit as a bot identity; push with rebase-on-conflict retry.
4. Wire it to a `personal-site-sync-X.{service,timer}` pair on the host where
   the data source lives (typically `cloud-hil-1`). See the books timer for
   the unit file shape.

## Conventions

- **Styling**: Tailwind v4 with custom `@utility` rules (`text-h1`, `text-h2`,
  etc.) defined in `src/styles/app.css`. Dark mode via `prefers-color-scheme`
  CSS variables вЂ” no JS toggle.
- **Icons**: add an SVG to `icons/`, rebuild the sprite with
  `npm run build:icons`, consume via `<Icon name="your-icon" />` which emits
  `<svg><use href="/sprite.svg#id" /></svg>`.
- **Interactivity**: prefer vanilla `<script>` blocks inside Astro components
  over React islands. Examples: `MobileNav.astro`, `Scrollspy.astro`.
- **SEO**: every page goes through `BaseLayout.astro`, which enforces `title`
  and `description` props at the type level. Canonical, OG, Twitter, and
  sitemap are handled there.
- **Blog**: drafts (`draft: true` in frontmatter) are visible in dev, hidden
  in prod. RSS at `/rss.xml`. Syntax highlighting via Shiki dual-theme.
- **Fonts**: self-hosted via `@fontsource` packages, imported in
  `BaseLayout.astro`.

## Deployment

Cloudflare Pages builds on push to `main`:
- Build command: `npm run build`
- Output dir: `dist`
- Node version: pinned via `.node-version`

Cache-Control overrides live in `public/_headers`.
